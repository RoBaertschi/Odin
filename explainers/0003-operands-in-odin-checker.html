Operand Means Checked Expression Result

In Odin's checker, an operand is the temporary semantic result produced while checking an expression. It answers: what did this syntax resolve to, what type does it have, and how may the surrounding code use it?

The Small Mental Model

An AST expression is syntax: x + 1, foo, int, a[i]. The checker walks that syntax and fills an Operand. The operand is not the syntax itself. It is the checker saying, "after name lookup, type construction, constant folding, and validity checks, this expression is a variable/value/constant/type/etc."

Why It Exists

A compiler constantly needs more than "this expression has type T". It also needs to know whether the expression is assignable, constant, a type expression, a built-in procedure, an overloaded procedure group, or an invalid expression that should suppress cascaded errors.

That extra classification is the operand's addressing mode.

The Actual Shape

struct Operand {
    AddressingMode mode;
    Type *         type;
    ExactValue     value;
    Ast *          expr;
    BuiltinProcId  builtin_id;
    Entity *       proc_group;
};

Source: src/checker.hpp

The Important Fields

mode
The category of result: value, variable, constant, type, no-value, invalid, and special Odin cases.
type
The semantic type attached to the result, such as int, ^T, or an untyped literal type.
value
The exact compile-time value, but only meaningful for constant operands.
expr
The source AST expression this operand came from, useful for diagnostics and recording semantic facts back onto the tree.

Addressing Mode Is The Key

Value
A computed result. You can read it, pass it, convert it, but it is not assignable as a storage location.
Variable
An addressable storage location. This is the closest Odin checker term to a C lvalue.
Constant
A compile-time known value. The checker can fold it and test representability in target types.
Type
The expression denotes a type, not a runtime value. This matters because Odin expressions can mention types in expression-like positions.
NoValue
The expression is valid but yields no value, like a void result in C terms.
Invalid
The checker found an error. Keeping an invalid operand lets later checker code continue without producing nonsense follow-up errors.

Example: Same Type, Different Operand

Imagine a C-like checker borrowing this model:

int x;
x;      // Variable operand, type int
x + 1;  // Value operand, type int
42;     // Constant operand, type untyped integer or int
int;    // Type operand, not a runtime value

The type alone cannot distinguish these. x, x + 1, and 42 may all be compatible with int, but only x can appear on the left side of assignment.

The Transferable Design

For your C compiler, an operand-like struct is the bridge between expression checking and statement checking. It should probably carry at least: validity, type, value category, constant value, source expression, and maybe symbol/entity reference.

Then assignment, calls, casts, unary operators, and binary operators can all consume one common semantic result type.
