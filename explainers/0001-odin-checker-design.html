Odin's Checker Is A Phased Semantic Pipeline

The transferable design is not "make type checking threaded". It is: split semantic analysis into phases, make declarations resolvable before bodies, and only parallelize work units whose shared state has already stabilized.

Core Model

Entity

A named semantic object. It carries kind, type, scope, state, flags, source order, and a link back to its declaration record.

Scope

A name-to-entity map with a parent. Package scopes, file scopes, type scopes, procedure scopes, and block scopes all use this idea.

Declaration Record

The unresolved payload for a declaration. It lets Odin collect names first, then resolve type expressions, initializers, signatures, and bodies later.

Operand

The result of expression checking. It is richer than a type: it records whether syntax produced a value, variable, constant, type, proc group, no-value, or invalid expression.

Pipeline

1
Create package and file scopesBuild the containers that declarations will be inserted into.
2
Collect entitiesWalk files, create entities and declaration records, insert names, enqueue unresolved entities.
3
Resolve imports and exportsOrder packages by import dependencies and make exported entities visible through package scopes.
4
Resolve global entitiesCheck constants, type names, global variables, procedure signatures, and procedure groups. Entity states catch declaration cycles.
5
Check procedure bodiesProcedure signatures are known, so bodies can be checked as mostly independent work items.
6
Run final semantic passesScope usage, type cycles, inline cycles, init order, dependency sets, entry point checks, and type-info tables.

Threading Rule

Odin uses parallelism where each worker can own a context and report results through queues. It uses phase barriers before later phases read those results as stable global state.

threaded:
  collect entities per file
  export entities per package
  check procedure bodies

serialized or barriered:
  import dependency order
  global entity resolution
  queue-to-array merging
  final whole-program checks
For a C compiler, the safest first transfer is single-threaded declaration and type resolution, followed by parallel function-body checking once all file-scope declarations are known.

C Compiler Transfer

parse translation units
create scopes
collect declarations
resolve file-scope declaration types
check global initializers
parallel check function bodies
merge diagnostics and semantic facts
run whole-program checks

The key structure to copy is the separation between a name-bearing entity and the declaration record that will later resolve it. That is what allows "declare first, check later" without losing source information.

Quick Check

Why does Odin check procedure bodies after resolving procedure declarations?

Because other code can call a procedure once its entity and signature are known. The body can be checked later as a separate work item, which makes dependency handling easier and creates a natural unit for parallel checking.
