An Entity Is Odin's Named Semantic Object

Read an entity as a durable semantic handle for a name. The parser gives Odin syntax; entity collection gives Odin named things that later phases can resolve, use, order, and check.

What It Is

Name plus kind

EntityKind says whether the name denotes a constant, variable, type name, procedure, procedure group, import name, library name, nil, label, or builtin.

Semantic state

EntityState records whether declaration resolution has not started, is in progress, or is resolved. This is central to declaration-cycle detection.

Scope membership

The entity points at its scope, and the scope maps interned names back to entities. Lookup gives the checker a semantic object rather than another AST search.

Declaration bridge

DeclInfo stores the AST payload needed to resolve the entity later: type expression, initializer, procedure literal, attributes, dependencies, and checked state.

Lifecycle

1. Collect File-scope declarations create placeholder entities and declaration records. Unresolved
2. Insert The entity is inserted into a scope and attached to its declaration record. Name visible
3. Queue Non-lazy entities are queued for declaration resolution after collection. Work item
4. Resolve check_entity_decl switches on kind and fills type, value, signature, group members, and flags. InProgress
5. Use Identifier checking maps the entity to an operand: constant, variable, type, procedure value, builtin, and so on. Resolved
The transferable pattern for a C compiler is to separate "the name exists" from "the declaration is fully checked." That lets declarations be collected first, then resolved in an order driven by dependencies.

Resolution path

check_entity_decl is the main dispatcher. It guards against cycles with EntityState_InProgress, then resolves based on entity kind.

Unresolved
  -> InProgress
    -> check_global_variable_decl
    -> check_const_decl
    -> check_type_decl
    -> check_proc_decl
    -> check_proc_group_decl
  -> Resolved

Expression path

check_ident uses the entity to produce an operand. The entity answers "what name is this?"; the operand answers "what did this expression produce?"

Mental Model

An entity is not just a symbol table row. In Odin it is the stable handle shared by declaration checking, expression checking, dependency tracking, documentation, visibility, code generation links, source ordering, and later whole-program passes.

That is why the struct looks busy: not because lookup itself is complicated, but because the same named object survives across phases.
