Odin Threads Procedure Bodies After Declaration Checking

Your model is mostly right: Odin resolves global declarations first, and procedure declarations queue body work for later. Then the queued bodies are checked by worker threads after a phase barrier.

The Shape

Collect Entities

Build named entities and scopes from parsed files.

Check Globals

Resolve declarations, types, constants, variables, and procedure signatures.

Queue Bodies

check_proc_decl stores ProcInfo instead of checking the block immediately.

Phase Barrier

Global declaration state is stable enough for body checking.

Check Bodies

Each worker checks procedure statements and expressions using the resolved signatures.

What Happens In A Procedure Declaration

check_proc_decl checks the procedure type, attributes, entry-point rules, calling convention, foreign/export constraints, and similar declaration-level facts. If the procedure has a body and is not an unspecialized polymorphic procedure, it calls check_procedure_later.

if (pl->body != nullptr) {
    d->scope = ctx->scope;
    if (!pt->is_polymorphic) {
        check_procedure_later(ctx->checker, ctx->file,
                              e->token, d, proc_type,
                              pl->body, pl->tags);
    }
}

The Design Rule

Do not type-check a procedure body while its surrounding world is still being discovered. First make names, signatures, and declaration records stable. Then body checking can run in parallel because each body mostly consumes that shared semantic state.

The Queue Switch

check_procedure_later has two modes. Before body checking begins, it appends to c->procs_to_check. During threaded body checking, newly discovered body work is submitted directly to the thread pool.

if (global_procedure_body_in_worker_queue.load()) {
    thread_pool_add_task(check_proc_info_worker_proc, info);
} else {
    array_add(&c->procs_to_check, info);
}

Source path: src/checker.cpp, check_procedure_later.

The Body Phase

check_procedure_bodies flips global_procedure_body_in_worker_queue, submits every queued ProcInfo as a worker task, clears the queue, waits for the thread pool, then flips the flag back.

global_procedure_body_in_worker_queue = true;

for_array(i, c->procs_to_check) {
    thread_pool_add_task(check_proc_info_worker_proc,
                         c->procs_to_check[i]);
}
array_clear(&c->procs_to_check);

thread_pool_wait();
global_procedure_body_in_worker_queue = false;

Source path: src/checker.cpp, check_procedure_bodies.

Why This Is Safe Enough

Signatures first
Calls can be checked because callee procedure types already exist.
Bodies later
Statement and expression checking is deferred until the checker has passed the global declaration phase.
Per-thread maps
Each worker has its own UntypedExprInfoMap, avoiding one obvious shared mutable hot spot.
MPSC queues
Workers can enqueue discovered entities and untyped-expression data, and later phases merge queues into arrays.
Proc checked state
proc_checked_mutex and proc_checked_state prevent two workers from checking the same body at once.

The Nuance

Odin is not simply "single-thread globals, threaded bodies." It also parallelizes some earlier collection/export/dependency work. But the transferable checker architecture is the phase split: declaration resolution creates stable semantic facts; body checking consumes them in parallel.

Transfer To A C Compiler

A C compiler can use the same shape:

collect declarations
resolve file-scope types, objects, and function signatures
queue function bodies
wait until global declarations are stable
parallel_for each queued function body:
    create thread-local checker context
    check statements and expressions
    enqueue discovered diagnostics / references / local metadata
merge worker queues
run final validation passes

The hard boundary is important: function bodies may refer to global functions, global variables, typedefs, enum constants, and struct tags. Those semantic objects should exist before body workers start.
