# Work-stealing CAS race resolution

The learner understands that Odin's work-stealing queue has a single owner for `bottom` and multiple possible contenders for `top`. Each thread has a ring-buffer-backed queue; submitted tasks go onto the current thread's queue, idle workers steal from other queues by compare-exchanging `top`, and the owner only needs compare-exchange on `top` for the last-task race. Only the contender that sees the expected `top` value and atomically advances it wins the task. The key insight is that a stale ring read is harmless if it is paired with a stale `top`, because the compare-exchange fails once another thread has advanced `top`. The useful mental model is "optimistic until validated": intermediate reads can be stale, but the CAS validates ownership before the read task is allowed to matter. The ring pointer is atomic for safe cross-thread publication/observation of the current ring buffer and its ordering, not because multiple threads resize the same queue.
